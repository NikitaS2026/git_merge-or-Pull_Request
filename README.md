Разница между git merge и Pull Request




__________НАПОМИНАЛКА ПО РАБОТЕ ПОЛНОСТЬЮ С ГИТХАБОМ___________
1. Начало работы (Ветки)

Всегда делай новую задачу в новой ветке, чтобы не испортить основную.

    git switch -c имя-ветки — создать новую ветку и перейти в неё (современный способ).

    git switch имя-ветки — перейти в существующую ветку.

    git branch — посмотреть список веток (где ты сейчас).

2. Сохранение и отправка (Commit & Push)

    git add . — подготовить все измененные файлы к сохранению.

    git commit -m "что именно сделано" — сохранить изменения (локально).

    git push -u origin имя-ветки — отправить ветку на GitHub в первый раз.

    git push — отправлять изменения в эту же ветку в следующие разы.

3. Слияние (Merge) и Pull Request

Когда задача готова и ты хочешь добавить её в main.

Способ А: В терминале (Merge) — для работы в одиночку.

    git switch main — идем в главную.

    git pull origin main — скачиваем свежее состояние.

    git merge имя-ветки — вливаем свою задачу в main.

    git push origin main — обновляем main на сервере.

Способ Б: На GitHub (Pull Request) — для работы в команде.

    Пушишь свою ветку.

    На сайте жмешь Compare & pull request.

    Выбираешь тип слияния:

        Create merge commit — сохранить все коммиты и создать запись о слиянии (классика).

        Squash and merge — «схлопнуть» 10 мелких коммитов в один чистый (лучший выбор для чистоты истории).

        Rebase and merge — выстроить коммиты в одну прямую линию (для идеального порядка).

4. Разрешение конфликтов

Если Git пишет "Conflict", ищи в файлах маркеры:

    <<<<<<< HEAD — твои текущие изменения.

    ======= — граница между версиями.

    >>>>>>> branch-name — изменения, которые ты вливаешь.

Что делать: Отредактируй файл (удали всё лишнее вместе с маркерами), затем:
git add . → git commit -m "Fix conflict" → git push.
5. Полезные команды (Tag и Отмена)

    Теги (Релизы):

        git tag -a v1.0 -m "Версия 1.0" — пометить важный момент (релиз).

        git push origin --tags — отправить теги на сервер.

    Отмена (Спасение):

        git restore имя-файла (или git checkout -- имя-файла) — вернуть файл к состоянию последнего коммита, если ты его случайно удалил или испортил.

6. Если push не работает (Rejected)

Если сервер не принимает код, значит там есть новые коммиты.

    git pull origin main — скачай новое.

    Исправь конфликты (если будут).

    Снова git push.

Золотое правило: Сначала git pull (скачай), потом работай, потом git push (отправляй).
